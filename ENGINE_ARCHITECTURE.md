# Архитектура Движка Диаграмм (Diagram Engine Architecture)

Этот документ описывает полный жизненный цикл построения диаграммы — от логической расстановки прямоугольников узлов до финальной векторной отрисовки ортогональных связей с мостиками на перекрестках.

---

## Единый Pipeline (Архитектурный инвариант)

Все типы диаграмм проходят через **идентичный** конвейер обработки. Процесс разбит на 4 последовательных этапа. Различается **только первый этап**, который уникален для каждой топологии.

`[Этап 1: Макро-компоновка] → [Этап 2: Распределение Портов] → [Этап 3: A* Маршрутизация] → [Этап 4: SVG-Вектор]`

1. **Макро-компоновка:** Зависит от `diagramType`. Для flowchart используется Dagre, для дерева — RT-алгоритм, для таймлайнов — Kahn top-sort. После вычисления координат, узлы жестко привязываются к сетке `20px`.
2. **Распределение Портов:** Каждой связи (до старта маршрутизации) выдаются выходы (Top/Root/Left/Right) с учетом матрицы геометрических штрафов.
3. **A* Routing:** Единый алгоритм прокладывает ортогональные ломаные линии по свободным клеткам сетки, огибая препятствия. Настраиваются только переменные: гравитация портов (вкл/выкл), busing (вкл/выкл). Исключение: radial использует прямые математические векторы без A*.
4. **SVG Постобработка:** Вставка инженерных разрывов-мостиков, скруглений углов (fillets) и специфичных маркеров.

## ЭТАП 1. Макро-компоновка и Топологии (Node Placement)

Задача этапа — рассчитать `(x, y)` всех узлов для минимизации пересечений.

### 1.1. Базовые правила
- **Единый вектор потока (Flow Direction):** Динамическая ориентация на основе габаритов холста была удалена для предсказуемости. Теперь генеральное направление жестко привязано к семантике `diagramType`:
  - **Вертикальный поток (Топ-Вниз, TB):** Деревья (`tree`) и Радиальные карты (`radial`).
  - **Горизонтальный поток (Слева-Направо, LR):** Блок-схемы (`flowchart`), ER-диаграммы (`erd`), Таймлайны и Матрицы.
- **Заморозка координат:** Если у ноды `lockPos === true`, она исключается из работы авто-алгоритма и сохраняет свои ручные координаты.
- **Резервирование места под текст (Text Spacing):** Связи с длинным текстом транслируются в невидимые "ноды-распорки", заставляя Dagre разъехаться на необходимую ширину еще до старта прокладки линий.
- **Швейцарский шаг (Snapping 20px):** Финальные координаты всегда кратны 20. Это абсолютный гарант того, что порты встанут идеально напротив друг друга по осям.

### 1.2. Спецификации по Типам Диаграмм (Topologies)

#### 1. Flowchart (Блок-схемы)
- **Компоновка:** Dagre (Sugiyama DAG) с активным **Happy Path (`weight=100`)**. Движок находит самый длинный логический путь DFS'ом и выстраивает его в идеальную прямую ось. Побочные обработчики циклов выдавливаются вбок.
- **Cycle-safe:** Обратные ребра обнаруживаются `color 0/1/2` алгоритмом и скрываются от Dagre, предотвращая разрыв слоев.
- **Swimlane-эвристика:** При малом числе узлов в группах движок автоматически строит горизонтальный поток, имитируя классические swimlane.
- **Зазоры:** `MIN_GAP = 40`. Сращивание запрещено (каждая связь индивидуальна).

#### 2. Tree / Hierarchy (Орг-структуры, Деревья)
- **Компоновка:** Custom Bottom-Up Engine (Reingold-Tilford). Инверсная логика осей по отношению к Flowchart.
- **Логика уровней:**
  - **Мульти-корни и Сироты:** Движок строит Лес из независимых деревьев, а "сирот" откладывает в матричный грид сбоку.
  - **Дети (Глубина 1):** Строятся всегда широкой горизонтальной строкой (до 10 в ряд). При превышении — шахматный порядок до 19.
  - **Внуки (Глубина 2+):** Упаковываются в вертикальные столбцы (до 5 блоков по высоте, максимум 3 столбца), чтобы крона не становилась бесконечно широкой. Для них активируется флаг `_stackEntry = 'Left'`.
- **Связи:** **Абсолютное доминирование Busing (`allowBusPremium = true`)**. Ветви слипаются в единый ствол (T-fork). Зазоры: `MIN_GAP = 80`.

#### 3. Sequence Diagram (Диаграммы последовательности)
- **Логика:** Увеличенные зазоры `MIN_GAP_X = 120` для lifelines. `MIN_GAP_Y = 80` определяет хронологический (вертикальный) шаг сообщения.
- **Связи:** Ортогональные Z-shape. Сращивание запрещено.

#### 4. Entity-Relationship (ERD & Архитектура)
- **Компоновка:** Dagre без Happy Path. Узлы распределяются "островками" с пространством `MIN_GAP = 80` для читаемости связей.
- **Связи:** A* маршрутизирует линии "как улочки", огибая прямоугольники. Свободный выбор портов со всех граней.

#### 5. Radial / Mind Map (Звёздные карты)
- **Логика:** Нода с максимальным `degree` становится центром. Остальные расходятся концентрическими кругами.
- **Компоновка:**
  - Угловой сектор детей пропорционален глубине их поддерева (дети = x0.4 веса, внуки = x0.1).
  - Сектор распространения сужается (±45° для детей, ±28° для внуков).
  - Применяется физический движок (20 итераций relaxation), рассталкивающий наложения по радиусу и касательной.
- **Связи (Direct-Line Routing):** Прямые линии через AABB/Овальные уравнения пересечений (без сетки).

#### 6. Timeline (Таймлайны)
- **Компоновка:** Топологическая сортировка (Kahn). Элементы расставляются змейкой, чередуя `+` и `-` смещение от базовой оси таймлайна (`crossOffset`). Сращивание запрещено. Увеличенные зазоры: 120/80.

#### 7. Matrix / Grid (Матрицы и квадранты)
- **Компоновка:** Сортировка по `groupId`. Модули разбиваются на ячейки `ceil(√N) × ceil(√M)`. Группы получают SVG-рамки.

## ЭТАП 2. Распределение Портов (Port Assigner)

Осуществляется через модуль `portAssigner.js` перед стартом маршрутизатора. Он формирует динамическую матрицу штрафов.

### Базовые штрафы (L-Rays Matrix)
Для ортогональных диаграмм проверяются все 4 грани (порты) узла.
- **Ideal Порт (Прямая видимость, `0 штрафа`):** L-луч (L-Ray) к цели не задевает чужие объекты.
- **Lateral Порт (Боковой, `1 × D штрафа`):** Требует обхода (где D — габарит узла: width или height).
- **Rear Порт (Задний, `2 × D штрафа`):** Требует огибания собственного узла.

### Специфические модификаторы
- **Растроение магистрали (Bifurcation penalty):** У нод с достаточными габаритами (`w >= 50px` или `h >= 50px`) добавляются по 2 запасных порта на соответствующих гранях (сдвиг ±20px от центральной оси). Изначально они облагаются штрафом `+2 × D`, чтобы использовались только при перегрузке центрального порта.
- **Диагональные порты круга:** Круги (`circle`) имеют запасные выходы под 45°, выведенные на 20px за границы. Штраф = диаметру.
- **Запрет смешения типов (Port Saturation Rule):** Запрещено выдавать один порт связям с разным визуальным стилем во избежание визуальной "грязи" (слияние пунктирной и сплошной линий). Если `edgeType` не совпадает, занятый порт получает блокирующий штраф `+10 × D`, заставляя A* увести штриховую линию в бифуркационный боковой порт.
- **Tree _stackEntry Перехват:** Для свернутых в вертикальные списки узлов (деревья глубины 2+) перехватывается флаг `_stackEntry = 'Left'`, обязывающий линию войти сбоку, а не пробивать блок навылет сверху вниз.

## ЭТАП 3. Ядро Маршрутизации (A* Routing & Penalties)

Сердце системы — `astar.js`. Каждая ломаная линия минимизирует функцию: `fScore = gScore (реальное расстояние) + hScore (Дистанция до safeTarget)`. 
Все узлы обернуты в **padBox** (+20px защитного поля), сквозь который прокладка линий запрещена (кроме собственных портов через `allowObsId`).

Очередь маршрутизации (Edge Prioritization) сортируется от коротких связей к длинным, чтобы локальные кластеры строились первыми, не пересекаясь с транзитными магистралями.

### 3.1 Каскадная Деградация (Tiers)
Если идеальный путь заблокирован, алгоритм поэтапно ослабляет ограничения:
- **Tiers 1-2 (Сетка 20px):** Идеал. Строгий запрет на наложение линий.
- **Tier 3 (Grid step 10px + Crossing):** Разрешено пробивать чужие линии под углом 90° (выдавая жесточайший Crossing Penalty). 
- **Tier 4 (Overlap Allowed):** Сетка 10px. Ослабляется запрет на наложение линий-сирот.
- **Fallback:** Векторы прокладываются напрямую.

### 3.2. Налоги и Запреты (Bans)
- **Изгиб (Bend Penalty = 100):** Налог на поворот на 90°. Вычищает Z-лесенки.
- **Обратный ход (Backtrack Penalty = 200):** Штраф за движение, уводящее от цели.
- **Транзитное пересечение (Crossing Penalty = 1500):** На Tier 3 за пробитие "чужой" линии.
- **Смертельный Запрет Слияния (Strict Bundling Ban):** Линии разной породы (`edgeType`) физически не имеют права слипаться даже при максимальной деградации и нехватке места. Разные сущности требуют разного воздуха.
- **Topological U-turn Ban:** Абсолютный запрет на поворот линии на 180° в пределах одной клетки.
- **Штраф за Текст (Text Abbreviation Penalty = 100):** Разовый штраф. Если на текущем пройденном участке маршрута нет ни одного прямого куска, способного вместить длину текста `edge.label`, на линию накладывается фиксированный эвристический штраф 100. A* всегда предпочтет маршрут с более длинным чистым прямым участком перегону с множеством коротких поворотов, но не будет сходить с ума и строить гигантские петли ради текста.

### 3.3. Премии и Дисконты (Discounts)
Механики, заставляющие A* принимать эстетичные "человечные" решения:
- **Z-выравнивание (Median Bend = 20):** Скидка за то, чтобы единственный Z-излом случился на строгой геометрической медиане между фигурами.
- **Т-перекрестки и Сращивание (T-Fork / Busing = 100):** Задействовано только для Орг-структуры (`tree`). Разрешает линиям накладываться в единую "Шину связи" (cost падает с 20 до 0.5 за шаг).

## ЭТАП 4. Векторная Постобработка и Слои (SVG)

Последний штрих перед рендерингом координат в React-компоненты:
1. Очистка колинеарных координат (удаление фальшивых проходных узлов).
2. **Скругление (Fillets):** На каждом макрометрическом изломе под 90° вычисляется квадратная кривая Безье (`Q`), делающая гладкий инженерный угол.
3. **Мостики пересечений (Jumps):** Сканируется таблица `ctx.occupiedLines`. Координаты из Tier 3 (пробитие чужой линии насквозь) получают микроразрыв — линия прерывается за несколько пикселей до пересечения и продолжается после него (команда `M` вместо дуги).

### Обработка текстовых надписей (Text Truncation Rules)
Постобработка и React-UI строго подчиняются трем законам геометрии текстов на линиях:
1. **Направление (Orientation Strictness):** Текст всегда читается слева направо (Left-to-Right). Для строго вертикальных линий — всегда снизу вверх (Bottom-to-Top). Никогда иначе.
2. **Нулевое Сращивание (No Bundle Zone):** Текст физически не отрисовывается на тех участках, где линия слипается с другой (bus overlaps). Только индивидуальные перегоны.
3. **Агрессивная Обрезка (Hard Truncate):** Если полный текст не помещается на максимальном доступном прямом отрезке (с учетом отступов под маркеры-стрелки ~15px), он **полностью скрывается**. Многоточия запрещены. Исключений нет (если только сам текст короче 3 символов, то он пытается уместиться математически).
4. **Исключение для Radial:** На радиальных диаграммах (Radial / Mind Map) текстовые подписи на связях принудительно отключены для визуальной чистоты, так как радиальные лучи обычно короткие, плотные и плохо подходят для текстов.

### Topology-Specific Визуальные Слои (`DiagramRenderer.jsx`)
- **Sequence (Lifelines):** Под каждую ноду опускается вертикальная пунктирная линия (`6, 4`) на 2000px вниз.
- **Matrix (Группы):** Вокруг групп ячеек рендерится пунктирная граница (`rx=8`) с именем группы в центре верхней стороны.
- **ERD (Crow's Foot):** На концы векторов (края связей) монтируются SVG маркеры. Вертикальная палка (`cf-one`) обозначает 1. "Воронья лапка" с 3 пальцами (`cf-many`) обозначает связь M:N.
